---
description: Analyze and understand PR review comments to learn from colleague feedback without making changes.
---

// turbo-all
1. Examine the provided screenshot or image of the PR comment to identify:
   - The code snippet that the reviewer is referring to
   - The specific comment or suggestion made by the reviewer
   - Any additional context (line numbers, file names, conversation thread)
2. Locate the corresponding code in the current workspace to understand:
   - The full context around the commented code section
   - The purpose and functionality of this code
   - How this code integrates with the rest of the codebase
   - The file path and location within the project architecture
3. Analyze the reviewer's comment to understand:
   - What specific aspect of the code they are addressing (performance, readability, architecture, security, best practices, etc.)
   - Whether it's a question, suggestion, or requirement
   - The severity or priority of the feedback (critical issue, improvement opportunity, minor suggestion)
4. Research the underlying concept or principle by:
   - Identifying the coding principle, pattern, or best practice being referenced
   - Understanding why this principle matters in this specific context
   - Exploring common pitfalls or issues the reviewer might be trying to prevent
   - Finding relevant language/framework-specific conventions or idioms
5. Evaluate the technical merit of the comment:
   - What problem could arise if the code remains unchanged
   - What benefits would the suggested approach provide (maintainability, performance, clarity, safety, etc.)
   - Are there trade-offs or alternative approaches to consider
   - How does this align with industry best practices or this project's standards
6. Extract learning opportunities by identifying:
   - Knowledge gaps this comment reveals (syntax, patterns, tools, practices)
   - Similar patterns in the codebase that could be reviewed
   - Preventive measures for future code (linting rules, design patterns, checklists)
   - Resources to study for deeper understanding (documentation, articles, examples)
7. Generate a comprehensive learning-focused explanation with the following sections:

**Comment Summary**
- Concise paraphrase of what the reviewer is saying
- The specific code pattern or element being discussed
- Classification of the feedback type (bug, optimization, style, architecture, question, etc.)

**Technical Context**
- Explanation of the current code approach and its intent
- Why the code was likely written this way initially
- The role of this code in the broader system

**Reviewer's Perspective**
- What the reviewer is trying to achieve or prevent with this comment
- The underlying principle, pattern, or best practice being referenced
- Why this matters in this specific context (performance, safety, maintainability, etc.)

**Technical Deep Dive**
- Detailed explanation of the concept or principle involved
- How the suggested approach differs from the current implementation
- Potential issues with the current code (if any)
- Benefits of the reviewer's suggestion
- Trade-offs or considerations to keep in mind

**Learning Takeaways**
- Key lessons to remember for future development
- Related patterns or practices to study
- How to recognize similar situations in the future
- Preventive strategies (tools, checklists, patterns) to apply

**Additional Context** (if applicable)
- Links to relevant documentation or standards
- Examples of similar implementations in the codebase
- Alternative approaches worth considering
- Questions to ask the reviewer for clarification

Remember: The goal is educational understanding, not implementation. Focus on extracting maximum learning value from the feedback.
